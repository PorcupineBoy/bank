## 1. 接口设计

### 1.1 设计约定

| 约定项 | 说明 |
|--------|------|
| 请求方式 | 所有接口统一使用POST |
| 请求格式 | Content-Type: application/json |
| 响应格式 | 统一JSON响应，详见《后端开发规范.md》第3章 |
| 认证方式 | 除注册/登录/发送验证码外，其余接口须在请求头携带JWT令牌 |
| 基类字段 | 所有请求DTO继承ReqBasic，自动注入tenant/userId/userName |

## 2. 数据库物理设计

### 2.1 设计规范

| 规范项 | 要求 |
|--------|------|
| 字符集 | utf8mb4 |
| 排序规则 | utf8mb4_unicode_ci |
| 主键策略 | BIGINT自增，命名 `xxx_id` |
| 时间戳 | `created_at` / `updated_at` 统一命名 |
| 敏感数据 | AES-256加密存储，禁止明文 |
| 状态字段 | TINYINT枚举，COMMENT说明取值含义 |
| 字段注释 | 所有字段必须添加中文COMMENT |
| 唯一索引 | 前缀 `uk_` |
| 非唯一索引 | 前缀 `idx_` |

### 2.2 表结构定义

#### 2.2.1 用户表（user）

| 字段名 | 类型 | 必填 | 默认值 | 索引 | 说明 |
|--------|------|------|--------|------|------|
| user_id | BIGINT | 是 | 自增 | PK | 用户唯一标识 |
| phone | VARCHAR(20) | 是 | — | uk_phone | 注册手机号 |
| password_hash | VARCHAR(255) | 是 | — | — | 登录密码bcrypt哈希 |
| trade_password_hash | VARCHAR(255) | 否 | NULL | — | 交易密码bcrypt哈希 |
| real_name | VARCHAR(50) | 是 | — | — | 真实姓名 |
| id_card_encrypted | VARCHAR(255) | 是 | — | — | 身份证号AES加密 |
| avatar_url | VARCHAR(500) | 否 | NULL | — | 头像地址 |
| nickname | VARCHAR(50) | 否 | NULL | — | 昵称 |

## 3. 核心服务设计

### 3.1 服务分层

后端服务采用经典的三层架构：

```
Controller层 ──> Service层 ──> Mapper/DAO层 ──> 数据库
    │              │
    ▼              ▼
RequestDTO    Business Logic      Entity/VO
(ReqBasic)    + 事务控制         (数据传输)
```

### 3.2 核心服务职责

| 服务 | 核心职责 |
|------|---------|
| AuthService | 注册/登录/登出、JWT管理、密码管理、验证码校验 |
| BankCardService | 绑卡/解卡、默认卡管理、余额查询、卡信息查询、四要素验证 |
| TransactionService | 转账、缴费、交易记录查询、限额校验、交易密码校验 |
| UserService | 用户信息查询与更新、头像管理 |
| SecurityService | 交易密码管理、限额查询与调整、密码错误计数 |

### 3.3 关键业务逻辑设计

**四要素验证（绑卡时）：**
```
1. 校验银行卡号Luhn算法
2. 校验持卡人姓名与注册用户real_name一致
3. 校验身份证号与注册用户id_card_encrypted解密后一致
4. 校验预留手机号格式及短信验证码有效性
5. 全部通过则允许绑定
```

**转账限额校验：**
```
1. 校验单笔限额：amount <= user.single_limit
2. 校验单日累计限额：user.daily_trans_amount + amount <= user.daily_limit
3. 校验付款卡余额（如对接核心系统）
4. 校验交易密码
5. 全部通过后扣减可用额度、创建交易记录
6. 交易成功后更新daily_trans_amount
```

**单设备登录控制：**
```
1. 用户登录成功后，生成JWT令牌
2. 将(userId, token)写入Redis，设置过期时间（如30分钟）
3. 同一用户新登录时，覆盖Redis中的token值
4. 旧token在Redis中失效，后续请求校验时拦截旧token
5. 旧设备收到401响应，自动跳转登录页
```

## 4. 数据流设计

### 4.1 转账数据流

```
[Client]
  │ POST /api/transactions/transfer
  ▼
[Nginx] ──> SSL终止、路由转发
  ▼
[BankController]
  │ 参数校验（@Validated）
  ▼
[TransactionService]
  │ 1. 查询付款卡信息
  │ 2. 校验交易密码
  │ 3. 校验单笔/单日限额
  │ 4. 校验余额（如对接核心）
  │ 5. 生成交易流水号
  │ 6. 创建交易记录（状态：处理中）
  │ 7. 调用核心系统/支付网关（异步）
  ▼
[TransactionMapper] ──> [MySQL: transaction表]
  │
  │ 交易结果回调
  ▼
[TransactionService]
  │ 更新交易状态（成功/失败）
  │ 记录操作日志
  ▼
[OperationLogMapper] ──> [MySQL: operation_log表]
  ▼
[Client] <── 返回处理结果
```

### 4.2 登录数据流

```
[Client]
  │ POST /api/auth/login/password
  ▼
[AuthController]
  ▼
[AuthService]
  │ 1. 查询用户（by phone）
  │ 2. 校验账户状态（非锁定/注销）
  │ 3. bcrypt校验密码
  │ 4. 密码错误则累加计数，达阈值则锁定
  │ 5. 密码正确则重置失败计数
  │ 6. 生成JWT令牌
  │ 7. 写入Redis（单设备控制）
  ▼
[Redis]
  ▼
[Client] <── 返回JWT令牌及用户信息
```
