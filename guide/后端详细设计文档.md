## 1. 接口设计

### 1.1 设计约定

| 约定项 | 说明 |
|--------|------|
| 请求方式 | 所有接口统一使用POST |
| 请求格式 | Content-Type: application/json |
| 响应格式 | 统一JSON响应，详见《后端开发规范.md》第3章 |
| 认证方式 | 除注册/登录/发送验证码外，其余接口须在请求头携带JWT令牌 |
| 基类字段 | 所有请求DTO继承ReqBasic，自动注入tenant/userId/userName |

### 1.3 接口详细定义

#### 1.3.1 认证模块

**接口1：发送验证码**

| 项 | 内容 |
|---|------|
| 路径 | POST /api/auth/sms/send |
| 说明 | 向指定手机号发送短信验证码 |

请求参数：
```json
{
  "phone": "string // 手机号，必填，11位"
}
```

响应示例：
```json
{
  "code": 200,
  "message": "验证码发送成功",
  "data": null,
  "timestamp": 1714300000000
}
```

## 3. 核心服务设计

### 3.1 服务分层

后端服务采用经典的三层架构：

```
Controller层 ──> Service层 ──> Mapper/DAO层 ──> 数据库
    │              │
    ▼              ▼
RequestDTO    Business Logic      Entity/VO
(ReqBasic)    + 事务控制         (数据传输)
```

### 3.2 核心服务职责

| 服务 | 核心职责 |
|------|---------|
| AuthService | 注册/登录/登出、JWT管理、密码管理、验证码校验 |
| BankCardService | 绑卡/解卡、默认卡管理、余额查询、卡信息查询、四要素验证 |
| TransactionService | 转账、缴费、交易记录查询、限额校验、交易密码校验 |
| UserService | 用户信息查询与更新、头像管理 |
| SecurityService | 交易密码管理、限额查询与调整、密码错误计数 |
| AiChatService | 智能客服对话、意图识别、Function Calling调度、聊天历史管理 |
| TransactionCategorizationService | 交易自动分类、消费分析统计、AI洞察生成、分类规则管理 |

### 3.3 关键业务逻辑设计

**四要素验证（绑卡时）：**
```
1. 校验银行卡号Luhn算法
2. 校验持卡人姓名与注册用户real_name一致
3. 校验身份证号与注册用户id_card_encrypted解密后一致
4. 校验预留手机号格式及短信验证码有效性
5. 全部通过则允许绑定
```

**转账限额校验：**
```
1. 校验单笔限额：amount <= user.single_limit
2. 校验单日累计限额：user.daily_trans_amount + amount <= user.daily_limit
3. 校验付款卡余额（如对接核心系统）
4. 校验交易密码
5. 全部通过后扣减可用额度、创建交易记录
6. 交易成功后更新daily_trans_amount
```

**单设备登录控制：**
```
1. 用户登录成功后，生成JWT令牌
2. 将(userId, token)写入Redis，设置过期时间（如30分钟）
3. 同一用户新登录时，覆盖Redis中的token值
4. 旧token在Redis中失效，后续请求校验时拦截旧token
5. 旧设备收到401响应，自动跳转登录页
```

### 3.4 AI Agent服务设计

#### 3.4.1 智能客服助手（AiChatService）

**设计目标：**
基于规则型意图识别实现无需外部LLM API的智能客服，降低合规风险与运营成本，同时预留LLM对接扩展点。

**意图识别引擎：**
```
输入：用户原始文本
  │
  ▼
[预处理] ──> 转小写、去除标点、提取数字/金额
  │
  ▼
[关键词匹配层] ──> 维护意图-关键词映射表
  │   ├── QUERY_BALANCE:   ["余额", "多少钱", "剩多少", "查余额"]
  │   ├── QUERY_TRANSACTIONS: ["交易", "明细", "记录", "转账记录"]
  │   ├── QUERY_CARDS:     ["银行卡", "我的卡", "卡片"]
  │   ├── TRANSFER_INTENT: ["转账", "汇款", "转给"]
  │   ├── CONSUMPTION_ANALYSIS: ["消费", "花了", "支出", "分析"]
  │   ├── GREETING:        ["你好", "您好", "hello", "hi"]
  │   └── HELP:            ["帮助", "怎么用", "功能"]
  │
  ▼
[Function Calling调度器]
  │   ├── QUERY_BALANCE ──> BankCardService.listCards(userId)
  │   ├── QUERY_TRANSACTIONS ──> TransactionService.queryTransactions(...)
  │   ├── QUERY_CARDS ──> BankCardService.listCards(userId)
  │   ├── CONSUMPTION_ANALYSIS ──> TransactionCategorizationService.analyzeMonthlyConsumption(...)
  │   └── TRANSFER_INTENT ──> 返回转账引导卡片（引导用户进入Transfer页面）
  │
  ▼
[回复生成器] ──> 将服务返回结果格式化为自然语言文本
  │
  ▼
输出：ChatMessageVO（含intent、functionCalled标记）
```

**会话管理：**
- session_id格式：`chat_{timestamp}_{userId末4位}`
- 历史记录持久化到`chat_message`表，按(user_id, session_id)查询
- 单用户支持多会话，前端可切换/新建会话

**安全与合规：**
- AI回复中涉及敏感数据（余额、卡号）遵循既有脱敏规则
- 转账类意图仅返回引导卡片，不直接执行转账操作
- 所有AI交互记录留痕，便于审计

## 4. 数据流设计

### 4.1 转账数据流

```
[Client]
  │ POST /api/transactions/transfer
  ▼

  │ 2. 校验交易密码
  │ 3. 校验单笔/单日限额
  │ 4. 校验余额（如对接核心）
  │ 5. 生成交易流水号
  │ 6. 创建交易记录（状态：处理中）
  │ 7. 调用核心系统/支付网关（异步）
  ▼
[TransactionMapper] ──> [MySQL: transaction表]
  │
  │ 交易结果回调
  ▼
[TransactionService]
  │ 更新交易状态（成功/失败）
  │ 记录操作日志
  ▼
[OperationLogMapper] ──> [MySQL: operation_log表]
  ▼
[Client] <── 返回处理结果
```

### 4.2 登录数据流

```
[Client]
  │ POST /api/auth/login/password
  ▼
[AuthController]
  ▼
[AuthService]
  │ 1. 查询用户（by phone）
  │ 2. 校验账户状态（非锁定/注销）
  │ 3. bcrypt校验密码
  │ 4. 密码错误则累加计数，达阈值则锁定
  │ 5. 密码正确则重置失败计数
  │ 6. 生成JWT令牌
  │ 7. 写入Redis（单设备控制）
  ▼
[Redis]
  ▼
[Client] <── 返回JWT令牌及用户信息
```
