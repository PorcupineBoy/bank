## 1. 引言

### 1.1 编写目的

本文档旨在对"简易手机银行核心业务功能"进行完整的需求分析与描述，作为后续系统设计、开发、测试及验收的基准依据。

### 1.2 项目背景

随着移动互联网的发展，个人用户对线上金融服务的需求日益增长。本项目旨在构建一套简易手机银行系统，涵盖账户登录、银行卡管理、交易查询、转账汇款、生活缴费等核心功能，并通过基础安全机制保障用户资金与信息安全，满足个人用户日常线上金融服务需求。

### 1.3 术语与缩写

| 术语/缩写 | 说明 |
|-----------|------|
| OTP | 一次性密码（One-Time Password） |
| JWT | JSON Web Token，用于身份认证的令牌机制 |
| KYC | 了解您的客户（Know Your Customer） |
| MFA | 多因素认证（Multi-Factor Authentication） |
| AES | 高级加密标准（Advanced Encryption Standard） |

- 《商业银行互联网银行管理暂行办法》
- OWASP Mobile Security Testing Guide
- 相关行业安全规范与标准

## 2. 项目概述

### 2.1 系统目标

构建一套功能完整、安全可靠的简易手机银行系统，使用户能够通过移动端完成日常金融操作，包括但不限于账户管理、资金查询、转账汇款和生活缴费。

### 2.2 用户角色

| 角色 | 描述 |
|------|------|
| 普通用户 | 手机银行的注册使用者，可使用全部客户端功能 |
| 系统管理员 | 负责后台配置、参数管理、风控规则维护 |

### 2.3 系统边界

本系统为简易版手机银行，包含以下核心模块：

1. **账户与登录模块** — 用户注册、登录认证、会话管理
2. **账户和银行卡管理模块** — 银行卡绑定、账户信息维护
3. **交易和查询模块** — 余额查询、账单查询、转账汇款、生活缴费
4. **基础安全模块** — 密码校验、限额控制、风险提示、二次确认、交易日志

### 2.4 运行环境

| 项目 | 要求 |
|------|------|
| 客户端 | Android 8.0+ / iOS 13.0+ |
| 后端服务 | Java/Spring Boot 或同等框架 |
| 数据库 | MySQL 8.0+ / PostgreSQL 14+ |
| 缓存 | Redis 6.0+ |
| 通信协议 | HTTPS (TLS 1.2+) |

### 2.6 Docker 部署说明

#### 2.6.1 MySQL 8.0 容器化部署

项目已提供两种 Docker 部署方式，均基于官方 `mysql:8.0` 镜像：

**方式一：docker-compose（推荐）**

项目根目录已放置 `docker-compose.yml`，执行以下命令一键启动：

```bash
docker-compose up -d
```

服务说明：
- 容器名：`bank-mysql`
- 端口映射：`3306:3306`
- 默认数据库：`bank_db`
- 默认 root 密码：`root`
- 业务账号：`bank_user` / `bank_pass`
- 首次启动时自动执行 `sql/init_schema.sql` 初始化表结构
- 数据持久化卷：`mysql_data`

**方式二：docker run 命令**

```bash
docker run -d \
  --name bank-mysql \
  -p 3306:3306 \
  -e MYSQL_ROOT_PASSWORD=root \
  -e MYSQL_DATABASE=bank_db \
  -e MYSQL_USER=bank_user \
  -e MYSQL_PASSWORD=bank_pass \
  -v mysql_data:/var/lib/mysql \
  -v ./sql/init_schema.sql:/docker-entrypoint-initdb.d/init_schema.sql \
  mysql:8.0 \
  --character-set-server=utf8mb4 \
  --collation-server=utf8mb4_unicode_ci \
  --default-authentication-plugin=mysql_native_password
```

**方式三：自定义 Dockerfile 构建**

项目根目录已放置 `Dockerfile.mysql`，执行以下命令构建并运行：

```bash
docker build -f Dockerfile.mysql -t bank-mysql .
docker run -d --name bank-mysql -p 3306:3306 bank-mysql
```

#### 2.6.2 前端工程规范

- `bank-frontend/.gitignore` 已配置，排除 `node_modules/`、`dist/`、日志文件及编辑器缓存，确保依赖目录不会被提交到版本库。
- 本地开发启动命令：
  ```bash
  cd bank-frontend
  npm install
  npm run serve
  ```
  默认访问地址：`http://localhost:8081/`

## 3. 功能性需求

### 3.1 模块一：账户与登录

#### 3.1.1 用户注册

| 项目 | 描述 |
|------|------|
| 功能编号 | F-AUTH-001 |
| 功能名称 | 用户注册 |
| 优先级 | P0（必须实现） |

**功能描述：**

用户通过手机号完成注册，绑定个人身份信息。

**详细需求：**

1. 用户输入手机号码，系统发送短信验证码（OTP），有效期5分钟。
2. 用户输入验证码完成手机号验证。
3. 用户设置登录密码，密码需满足以下规则：
   - 长度8~20位
   - 须包含大写字母、小写字母、数字中的至少两种
   - 不得包含连续3位及以上的相同字符
4. 用户填写姓名、身份证号（用于基础KYC校验）。
5. 注册成功后生成唯一用户ID，跳转至登录页。

**输入/输出：**

| 输入 | 输出 |
|------|------|
| 手机号、验证码、登录密码、姓名、身份证号 | 注册成功/失败提示；成功则跳转登录页 |

**业务规则：**

- 同一手机号仅能注册一个账户。
- 短信验证码60秒内不可重复发送。
- 验证码输入错误超过5次，锁定该手机号注册操作30分钟。

---

#### 3.1.2 用户登录

| 项目 | 描述 |
|------|------|
| 功能编号 | F-AUTH-002 |
| 功能名称 | 用户登录 |
| 优先级 | P0 |

**功能描述：**

用户通过手机号+密码或手机号+短信验证码完成登录认证。

**详细需求：**

1. **密码登录：** 用户输入手机号和登录密码，系统校验通过后签发JWT令牌。
2. **验证码登录：** 用户输入手机号，获取短信验证码，输入验证码完成登录。
3. 登录成功后进入首页，展示账户概览信息。

**业务规则：**

- 密码连续输入错误5次，账户锁定30分钟。
- 登录令牌（JWT）有效期为30分钟，支持自动续期（sliding window）。
- 单设备登录限制：同一账户同一时间仅允许在一台设备上保持登录状态，新设备登录将使旧设备自动下线。

---

#### 3.1.3 会话管理

| 项目 | 描述 |
|------|------|
| 功能编号 | F-AUTH-003 |
| 功能名称 | 会话与状态管理 |
| 优先级 | P0 |

**详细需求：**

1. 用户登录后维持会话状态，通过JWT进行身份校验。
2. 用户无操作超过15分钟，自动退出登录并跳转至登录页。
3. 用户可主动点击"退出登录"结束会话。
4. 退出登录后清除本地令牌和敏感缓存数据。

---

#### 3.1.4 密码管理

| 项目 | 描述 |
|------|------|
| 功能编号 | F-AUTH-004 |
| 功能名称 | 密码找回与修改 |
| 优先级 | P1 |

**详细需求：**

1. **忘记密码：** 用户通过手机号+短信验证码+身份证后四位验证身份后，可重置登录密码。
2. **修改密码：** 已登录用户可在"安全设置"中修改登录密码，需输入原密码验证。
3. 新密码不得与最近3次使用过的密码相同。

### 3.2 模块二：账户和银行卡管理

#### 3.2.1 银行卡绑定

| 项目 | 描述 |
|------|------|
| 功能编号 | F-CARD-001 |
| 功能名称 | 银行卡绑定 |
| 优先级 | P0 |

**功能描述：**

用户绑定本人名下的银行借记卡或信用卡至手机银行账户。

**详细需求：**

1. 用户输入银行卡号，系统自动识别发卡行及卡类型（借记卡/信用卡）。
2. 用户输入持卡人姓名、身份证号、银行预留手机号。
3. 系统向银行预留手机号发送验证码，用户输入验证码完成绑定。
4. 绑定成功后，银行卡信息（脱敏展示）出现在"我的银行卡"列表中。

**业务规则：**

- 每个账户最多绑定10张银行卡。
- 银行卡号需通过Luhn算法校验。
- 持卡人姓名须与注册用户姓名一致（四要素验证）。
- 同一张卡不可重复绑定。

---

#### 3.2.2 银行卡管理

| 项目 | 描述 |
|------|------|
| 功能编号 | F-CARD-002 |
| 功能名称 | 银行卡查看与解绑 |
| 优先级 | P0 |

**详细需求：**

1. **查看银行卡列表：** 展示用户已绑定的所有银行卡，卡号脱敏显示（仅显示后四位）。
2. **查看卡片详情：** 点击单张卡片可查看发卡行、卡类型、绑定时间等信息。
3. **解绑银行卡：** 用户可解绑已绑定的银行卡，解绑前需输入登录密码确认。
4. **设置默认卡：** 用户可设置一张卡为默认付款卡。

**业务规则：**

- 若该卡有进行中的交易（如待处理的转账），不允许解绑。
- 解绑后该卡的历史交易记录保留，但不可再用于新交易。

### 3.3 模块三：交易和查询

#### 3.3.1 余额查询

| 项目 | 描述 |
|------|------|
| 功能编号 | F-TRANS-001 |
| 功能名称 | 账户余额查询 |
| 优先级 | P0 |

**详细需求：**

1. 首页展示用户已绑定银行卡的总资产概览（可选择隐藏金额）。
2. 点击具体银行卡可查询该卡的实时余额（向银行系统发起查询）。
3. 支持"一键隐藏/显示"余额功能（眼睛图标切换）。

**业务规则：**

- 余额查询需登录态有效。
- 查询失败时展示友好提示，并提供重试入口。

#### 3.3.2 账单查询

| 项目 | 描述 |
|------|------|
| 功能编号 | F-TRANS-002 |
| 功能名称 | 交易账单查询 |
| 优先级 | P0 |

**详细需求：**

1. 用户可按银行卡筛选查看交易明细。
2. 支持按时间范围筛选：近7天、近1个月、近3个月、自定义时间段。
3. 支持按交易类型筛选：全部、收入、支出、转账、缴费。

5. 交易明细列表支持下拉刷新和上滑分页加载。
6. 点击单条记录可查看交易详情（包含完整交易流水号等）。

**业务规则：**

- 默认展示近1个月的交易记录。
- 单次查询最多返回500条记录。
- 自定义时间段最大跨度为12个月。

#### 3.3.3 转账汇款

| 项目 | 描述 |
|------|------|
| 功能编号 | F-TRANS-003 |
| 功能名称 | 转账汇款 |
| 优先级 | P0 |

**功能描述：**

用户可从已绑定的银行卡向其他银行账户进行转账。

**详细需求：**

1. 用户选择付款银行卡（默认选中默认卡）。
2. 用户输入收款人信息：
   - 收款人姓名
   - 收款银行卡号
   - 收款银行（可通过卡号自动识别或手动选择）
3. 用户输入转账金额和转账备注（可选）。
4. 系统展示转账确认页面，包含：付款卡号（脱敏）、收款人信息、转账金额、手续费（如有）。
5. 用户输入交易密码进行二次确认。
6. 系统发起转账请求，展示处理结果（成功/处理中/失败）。

**业务规则：**

- 单笔转账限额：默认5万元，用户可在安全设置中调整（最高50万元）。
- 单日累计转账限额：默认20万元。
- 转账金额精度为分（0.01元），最小转账金额为0.01元。
- 收款卡号需通过Luhn算法校验。
- 转账处理中的交易需支持状态查询。
- 同一收款方连续转账间隔不少于30秒（防重复提交）。

**异常处理：**

| 异常场景 | 处理方式 |
|---------|---------|
| 余额不足 | 提示余额不足，建议更换付款卡 |
| 超过单笔限额 | 提示超出限额，展示当前限额值 |
| 超过单日累计限额 | 提示超出当日限额，展示剩余可用额度 |
| 网络异常 | 提示网络异常，建议稍后重试并查询交易状态 |
| 收款账户信息有误 | 提示信息有误，建议核实后重试 |

#### 3.3.4 生活缴费

| 项目 | 描述 |
|------|------|
| 功能编号 | F-TRANS-004 |
| 功能名称 | 生活缴费 |
| 优先级 | P1 |

**功能描述：**

用户可通过手机银行完成水费、电费、燃气费、话费等日常生活缴费。

**详细需求：**

1. 系统展示缴费类型列表：水费、电费、燃气费、话费充值。
2. 用户选择缴费类型后，输入对应的缴费户号/手机号。
3. 系统查询待缴费用（对接缴费平台接口）。
4. 用户确认缴费金额，选择付款银行卡。
5. 用户输入交易密码完成缴费。
6. 展示缴费结果。

**业务规则：**

- 缴费记录计入交易明细。
- 缴费限额与转账限额独立计算。
- 单笔缴费金额上限为1万元。
- 支持缴费户号/手机号的保存和历史记录，方便下次快速缴费。

## 5. 数据字典

### 5.0 数据库设计规范

1. **字符集**：统一使用 `utf8mb4`，排序规则 `utf8mb4_unicode_ci`。
2. **主键策略**：所有业务表主键均为 `BIGINT` 自增，名称为 `xxx_id`。
3. **时间戳字段**：
   - 创建时间统一命名为 `created_at`，默认 `CURRENT_TIMESTAMP`。
   - 更新时间统一命名为 `updated_at`，默认 `CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP`（视表需求而定）。
4. **敏感数据存储**：身份证号、银行卡号等敏感字段须使用 AES-256 加密存储，禁止明文保存；页面展示使用脱敏值。
5. **状态字段**：使用 `TINYINT` 枚举，需在注释中说明各取值的业务含义。
6. **字段中文注释**：**所有表字段必须添加 `COMMENT '中文注释'`**，注释内容须与数据字典保持一致，便于 DBA、运维及后续开发人员快速理解表结构。
7. **索引规范**：
   - 业务唯一约束使用 `UNIQUE KEY`，命名前缀 `uk_`。
   - 非唯一索引命名前缀 `idx_`。

## 7. 安全性需求

### 7.1 数据安全

1. **传输安全：** 客户端与服务端通信强制使用HTTPS（TLS 1.2+），禁止HTTP明文传输。
2. **存储安全：**
   - 密码使用bcrypt（cost >= 12）加盐哈希存储。
   - 银行卡号、身份证号使用AES-256-GCM加密存储。
   - 密钥管理采用独立的密钥管理服务（KMS）。
3. **展示安全：** 前端页面中银行卡号、身份证号、手机号等敏感信息一律脱敏展示。

### 7.2 接口安全

1. 所有API需验证JWT令牌有效性。
2. 接口请求需携带时间戳与签名，防止重放攻击。
3. 敏感操作接口（转账、缴费、修改密码等）需限流处理。
4. 使用参数化查询防止SQL注入。
5. 输入输出做XSS过滤。

### 7.3 审计与合规

1. 所有关键操作留痕（操作日志），日志不可篡改。
2. 日志保留不少于5年。
3. 日志中不得记录明文密码、完整卡号等敏感信息。

### 8.2 核心业务流程图（文字描述）

**转账业务流程：**

```
用户发起转账
  -> 选择付款银行卡
  -> 输入收款人信息与金额
  -> 系统校验：卡号格式、限额、余额
     -> 校验不通过 -> 提示异常原因
     -> 校验通过 -> 展示确认页面

           -> 密码错误 -> 提示错误（累计计数）
           -> 密码正确 -> 提交转账请求
              -> 记录交易日志
              -> 返回转账结果（成功/处理中/失败）
```

**登录业务流程：**

```
用户选择登录方式
  -> 密码登录：输入手机号+密码
     -> 校验通过 -> 签发JWT -> 进入首页
```
