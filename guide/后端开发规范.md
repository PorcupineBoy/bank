## 1. Controller 层入参规范

### 1.1 入参基类 ReqBasic

**所有 Controller 接口的入参 DTO 均须继承基类 `ReqBasic`**，基类定义如下：

```java
package com.bank.dto;

import lombok.Data;
import java.io.Serializable;

@Data
public class ReqBasic implements Serializable {
    private static final long serialVersionUID = 1L;

    /** 租户标识 */
    private String tenant;

    /** 用户ID（由 JWT 拦截器自动注入） */
    private Long userId;

    /** 用户名（由请求头或 JWT 自动注入） */
    private String userName;
}
```

**基类字段说明：**

| 字段 | 类型 | 来源 | 说明 |
|------|------|------|------|
| tenant | String | 请求头 `tenant` | 租户标识，多租户场景下使用 |
| userId | Long | JWT 拦截器自动注入 | 当前登录用户 ID，无需前端传参 |
| userName | String | 请求头 `userName` | 当前登录用户名 |

### 1.2 DTO 继承规范

- 所有业务入参类 **必须继承 `ReqBasic`**，并在此基础上定义业务字段
- 前端 **无需** 传递 `tenant`、`userId`、`userName`，由 `ReqBasicAdvice`（`RequestBodyAdvice` 实现）自动从 JWT 拦截器属性和请求头注入
- 未登录接口（注册、登录、发送验证码等）的入参 DTO 同样继承 `ReqBasic`，此时 `userId` 为 `null`

**示例：**

```java
// 正确 ✓
@Data
public class CardBindRequest extends ReqBasic {
    @NotBlank(message = "Card number is required")
    private String cardNo;
    // ...其他业务字段
}

// 错误 ✗ — 未继承 ReqBasic
@Data
public class CardBindRequest {
    private String cardNo;
}
```

### 1.3 Controller 层获取用户 ID

- 通过 `@RequestBody` 接收的 DTO，直接调用 `request.getUserId()` 获取用户 ID
- **禁止** 在 Controller 中通过 `HttpServletRequest.getAttribute("userId")` 获取用户 ID（无 `@RequestBody` 的接口除外）

```java
// 正确 ✓
@PostMapping("/bind")
public Result<Void> bindCard(@RequestBody @Validated CardBindRequest request) {
    bankCardService.bindCard(request.getUserId(), request);
    return Result.success();
}

// 错误 ✗ — 不应再通过 HttpServletRequest 获取 userId
@PostMapping("/bind")
public Result<Void> bindCard(@RequestBody @Validated CardBindRequest request, HttpServletRequest httpRequest) {
    Long userId = (Long) httpRequest.getAttribute("userId");
    bankCardService.bindCard(userId, request);
    return Result.success();
}
```

### 1.4 禁止使用 Map 作为入参

- Controller 层 **禁止** 使用 `Map<String, Object>`、`Map<String, String>`、`Map<String, Long>` 等作为 `@RequestBody` 入参
- 必须定义对应的 DTO 类继承 `ReqBasic`，确保入参类型安全、可校验、可文档化

```java
// 正确 ✓
@PostMapping("/detail")
public Result<BankCardVO> getCardDetail(@RequestBody @Validated CardIdRequest request) {
    return Result.success(bankCardService.getCardDetail(request.getUserId(), request.getCardId()));
}

// 错误 ✗ — 使用 Map 作为入参
@PostMapping("/detail")
public Result<BankCardVO> getCardDetail(@RequestBody Map<String, Long> body, HttpServletRequest request) {
    Long userId = (Long) request.getAttribute("userId");
    Long cardId = body.get("cardId");
    return Result.success(bankCardService.getCardDetail(userId, cardId));
}
```

## 2. 统一请求方式

- 所有 API 接口统一使用 **POST** 请求方式（禁止 GET / PUT / DELETE），提升接口安全性
- 前端 API 层须同步适配 POST 请求

## 4. 数据库设计规范

1. **字符集**：统一使用 `utf8mb4`，排序规则 `utf8mb4_unicode_ci`。
2. **主键策略**：所有业务表主键均为 `BIGINT` 自增，名称为 `xxx_id`。
3. **时间戳字段**：
   - 创建时间统一命名为 `created_at`，默认 `CURRENT_TIMESTAMP`。
   - 更新时间统一命名为 `updated_at`，默认 `CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP`（视表需求而定）。
4. **敏感数据存储**：身份证号、银行卡号等敏感字段须使用 AES-256 加密存储，禁止明文保存；页面展示使用脱敏值。
5. **状态字段**：使用 `TINYINT` 枚举，需在注释中说明各取值的业务含义。
6. **字段中文注释**：**所有表字段必须添加 `COMMENT '中文注释'`**，注释内容须与数据字典保持一致，便于 DBA、运维及后续开发人员快速理解表结构。
7. **索引规范**：
   - 业务唯一约束使用 `UNIQUE KEY`，命名前缀 `uk_`。
   - 非唯一索引命名前缀 `idx_`。

## 5. 后端项目结构规范

后端基于 Spring Boot + MyBatis Plus 构建，核心目录结构如下：

```
bank-backend/
├── src/main/java/com/bank/
│   ├── aspect/           # AOP 切面（操作日志等）
│   ├── common/           # 公共常量、统一响应封装
│   ├── config/           # 框架配置（MyBatisPlus、Redis、Web、ReqBasicAdvice）
│   ├── controller/       # REST API 控制器层
│   ├── dto/              # 请求/响应数据传输对象（均继承 ReqBasic）
│   ├── entity/           # 数据库实体类
│   ├── exception/        # 业务异常与全局异常处理
│   ├── interceptor/      # JWT 登录拦截器
│   ├── mapper/           # MyBatis Mapper 接口
│   ├── service/          # 业务逻辑层（接口 + 实现）
│   ├── util/             # 工具类（AES、JWT、Luhn、密码）
│   ├── vo/               # 视图层返回对象
│   └── BankApplication.java
├── src/main/resources/
│   ├── mapper/           # MyBatis XML 映射文件
│   └── application.yml   # 应用配置文件
└── pom.xml               # Maven 依赖管理
```

## 6. 安全规范

### 6.1 数据安全

1. **传输安全：** 客户端与服务端通信强制使用 HTTPS（TLS 1.2+），禁止 HTTP 明文传输。
2. **存储安全：**
   - 密码使用 bcrypt（cost >= 12）加盐哈希存储。
   - 银行卡号、身份证号使用 AES-256-CBC 加密存储。
   - 密钥管理采用独立的密钥管理服务（KMS）。
3. **展示安全：** 前端页面中银行卡号、身份证号、手机号等敏感信息一律脱敏展示。

### 6.2 接口安全

1. 所有 API 需验证 JWT 令牌有效性。
2. 接口请求需携带时间戳与签名，防止重放攻击。
3. 敏感操作接口（转账、缴费、修改密码等）需限流处理。
4. 使用参数化查询防止 SQL 注入。
5. 输入输出做 XSS 过滤。

### 6.3 审计与合规

1. 所有关键操作留痕（操作日志），日志不可篡改。
2. 日志保留不少于 5 年。
3. 日志中不得记录明文密码、完整卡号等敏感信息。
