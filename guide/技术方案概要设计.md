## 1. 设计约束与目标

### 1.1 设计约束

| 约束类型 | 约束内容 |
|---------|---------|
| 法规约束 | 须符合《个人金融信息保护技术规范》《移动金融客户端安全管理规范》等金融行业监管要求 |
| 安全约束 | 全链路HTTPS、敏感数据加密存储、密码加盐哈希、操作日志不可篡改 |
| 兼容性约束 | 支持Android 8.0+ / iOS 13.0+ |
| 性能约束 | 登录<=2秒、余额查询<=3秒、转账<=5秒、可用性>=99.9% |
| 运维约束 | 支持容器化部署、健康检查、配置外部化 |

### 1.2 设计目标

1. **高安全性**：多层安全防护体系，满足金融行业基础安全基线
2. **高可用性**：关键服务无单点故障，支持故障自动转移
3. **可扩展性**：业务模块松耦合，支持水平扩展
4. **可维护性**：统一技术栈、标准化工程结构、自动化运维支持

## 2. 系统总体架构

### 2.1 架构概述

系统采用经典的分层架构，自上而下分为：客户端层、接入层、应用服务层、数据层。各层职责清晰，通过标准协议交互。

### 2.3 模块职责

| 模块 | 职责描述 |
|------|---------|
| 认证服务 | 用户注册、登录、登出、JWT令牌签发与校验、短信验证码管理 |
| 银行卡服务 | 银行卡绑定、解绑、默认卡设置、卡信息查询、余额查询 |
| 交易服务 | 转账汇款、生活缴费、交易记录查询、收款方管理 |
| 安全服务 | 交易密码管理、限额控制、登录/交易密码错误计数、风控规则执行 |
| 用户服务 | 用户信息查询与更新、头像管理、手机号变更 |
| 日志服务 | 操作日志记录、审计日志查询（内部使用） |

## 4. 部署架构

### 4.1 开发/测试环境部署

开发及测试环境采用Docker Compose进行容器化编排，确保环境一致性。

**基础设施服务：**

| 服务 | 容器名 | 端口映射 | 说明 |
|------|--------|---------|------|
| MySQL | bank-mysql | 3306:3306 | 业务数据库，自动执行初始化脚本 |
| Redis | bank-redis | 6379:6379 | 会话缓存、验证码存储、限流计数 |

**启动方式：**
```bash
docker compose up -d
```

**数据持久化：**
- MySQL数据卷：`mysql_data`
- Redis数据卷：`redis_data`（AOF持久化开启）

### 4.2 生产环境部署建议

| 组件 | 部署建议 |
|------|---------|
| 应用服务 | 多实例部署（>=2节点），前置负载均衡 |
| MySQL | 主从架构，读写分离，定期自动备份 |
| Redis | 哨兵模式或集群模式，保障高可用 |
| Nginx | 双节点热备，SSL证书自动续期 |
| 监控 | 接入Prometheus + Grafana，关键指标告警 |

### 4.3 配置外部化

生产环境所有敏感配置（数据库密码、Redis密码、JWT密钥等）须通过环境变量或配置中心注入，禁止硬编码于配置文件或镜像中。

## 5. 安全架构

### 5.1 安全分层模型

```
┌─────────────────────────────────────────┐
│           应用层安全                      │
│  - 身份认证（JWT）                        │
│  - 权限校验                              │
│  - 输入校验与参数化查询                    │
│  - XSS/SQL注入防护                        │
├─────────────────────────────────────────┤
│           传输层安全                      │
│  - 全链路HTTPS（TLS 1.2+）               │
│  - HSTS、证书固定                        │
├─────────────────────────────────────────┤
│           数据层安全                      │
│  - 密码bcrypt加盐哈希                    │
│  - 敏感数据AES-256加密                   │
│  - 数据库访问账号最小权限                  │
├─────────────────────────────────────────┤
│           审计层安全                      │
│  - 操作日志完整留痕                       │
│  - 日志不可篡改                          │
│  - 敏感信息日志脱敏                       │
└─────────────────────────────────────────┘
```

### 5.2 身份认证机制

采用JWT（JSON Web Token）进行无状态身份认证：

1. 用户登录成功后，服务端签发JWT令牌（有效期30分钟）
2. 客户端后续请求在Header中携带令牌
3. 服务端拦截器验证令牌有效性（签名、过期时间）
4. 支持滑动窗口续期：活跃用户在令牌过期前自动续签
5. 单设备登录：新登录使旧令牌失效（Redis记录有效令牌黑名单）

### 5.3 数据加密策略

| 数据类型 | 存储策略 | 算法 |
|---------|---------|------|
| 登录密码 | 加盐哈希 | bcrypt（cost >= 12） |
| 交易密码 | 加盐哈希 | bcrypt（cost >= 12） |
| 身份证号 | 对称加密 | AES-256-CBC |
| 银行卡号 | 对称加密 | AES-256-CBC |
| 卡号脱敏值 | 明文存储（仅展示用） | — |

**密钥管理：** 开发测试环境使用配置文件管理密钥；生产环境须接入KMS（密钥管理服务）。

### 5.4 防攻击机制

| 攻击类型 | 防护措施 |
|---------|---------|
| 暴力破解 | 密码错误次数锁定（登录5次/30分钟，交易密码5次/24小时） |
| 短信轰炸 | 验证码发送频率限制（60秒间隔，同号码日上限10条） |
| 重放攻击 | 接口请求携带时间戳与随机数，服务端校验时效性 |
| SQL注入 | ORM框架参数化查询，禁止SQL拼接 |
| XSS攻击 | 输入过滤与输出转义 |
| CSRF攻击 | 同源校验 + Token机制 |

## 6. 数据架构

### 6.1 数据分层

| 层级 | 说明 | 对应存储 |
|------|------|---------|
| 源数据层 | 业务操作产生的原始数据 | MySQL |
| 缓存层 | 会话状态、热点数据、限频计数 | Redis |
| 日志层 | 操作审计日志 | MySQL（独立表或归档库） |

### 6.2 核心实体关系

```
┌──────────────┐       ┌──────────────┐       ┌──────────────┐
│    User      │◄──────┤  BankCard    │       │ Transaction  │
│   (用户)      │  1:n  │   (银行卡)    │       │  (交易记录)   │
└──────────────┘       └──────────────┘       └──────┬───────┘
       ▲                                              │
       │ 1:n                                          │ n:1
       │                                              │
┌──────┴───────┐       ┌──────────────┐              │
│ FrequentPayee│       │BillPayment   │◄─────────────┘
│ (常用收款方)  │       │  (缴费记录)   │
└──────────────┘       └──────────────┘
```

**关系说明：**
- 一个用户可绑定多张银行卡（1:n）
- 一个用户可有多条交易记录（1:n）
- 一笔交易对应一条缴费记录（1:1，缴费类型交易时）
- 一个用户可保存多个常用收款方（1:n）

### 6.3 数据生命周期

| 数据类型 | 保留策略 |
|---------|---------|
| 用户账户数据 | 账户注销后匿名化保留180天，之后删除 |
| 交易记录 | 永久保留（金融法规要求） |
| 操作日志 | 保留不少于5年 |
| 短信验证码记录 | 保留7天后自动清理 |
| 会话令牌 | 过期后自动清理 |

## 7. 集成架构

### 7.1 外部系统集成

| 外部系统 | 集成方式 | 用途 |
|---------|---------|------|
| 短信网关 | HTTP API | 发送注册/登录/交易验证码 |
| 银行核心系统 | 专线/HTTPS API | 余额查询、转账指令下发 |
| 缴费平台 | HTTPS API | 水/电/燃气/话费账单查询与缴费 |
| 银联/网联 | 专线 | 跨行转账清算 |

### 7.2 集成安全要求

1. 所有外部系统通信须使用HTTPS或专线
2. 接口调用须携带身份认证凭证（API Key / 数字证书）
3. 敏感接口增加签名机制，防止请求篡改
4. 外部接口调用失败须有降级策略和重试机制（有限重试，避免雪崩）

## 8. 性能与扩展性设计

### 8.1 性能优化策略

| 优化点 | 策略 |
|--------|------|
| 数据库查询 | 合理索引设计、分页查询、避免N+1查询 |
| 缓存使用 | 用户会话Redis存储、热点数据缓存、验证码TTL管理 |
| 接口响应 | 异步处理非关键路径（如日志写入）、连接池优化 |
| 前端加载 | 路由懒加载、组件按需加载、静态资源CDN加速 |

### 8.2 扩展性设计

1. **应用层水平扩展**：应用服务无状态化，支持多节点部署，通过负载均衡分发流量
2. **数据库扩展**：读多写少场景支持读写分离；未来可考虑分库分表（按user_id哈希）
3. **缓存扩展**：Redis支持集群模式，满足会话量增长需求
4. **服务拆分**：当前单体架构可平滑演进为微服务（认证服务、交易服务独立部署）

### 8.3 容量预估（初期）

| 指标 | 预估 |
|------|------|
| 注册用户 | 10万 |
| 日活跃用户 | 1万 |
| 日交易笔数 | 5,000 |
| 数据存储增长 | ~10GB/年（不含日志归档） |
